La Ingeniería de Requisitos es la disciplina encargada de identificar, analizar, documentar, validar y gestionar los requisitos de un sistema de software. Esta guía cubre los temas del 1 al 4 con definiciones claras y ejemplos prácticos.

Tema 1 — Introducción a la Ingeniería de Requisitos

¿Qué es un requisito?

Definición: Es una necesidad, condición o capacidad que debe cumplir un sistema para satisfacer las necesidades de los usuarios o del negocio.
Ejemplo
Una tienda virtual necesita que sus clientes puedan comprar en línea.

Requisito El sistema debe permitir registrar pedidos de compra.

¿Qué es la Ingeniería de Requisitos?

Definición: Es la disciplina encargada de identificar, analizar, documentar, validar y gestionar los requisitos de un sistema de software.

Objetivos

Ejemplo — Objetivos principales
  • Comprender las necesidades del cliente.
  • Evitar errores durante el desarrollo.
  • Reducir costos y retrabajos.
  • Garantizar que el software cumpla los objetivos del negocio.

Importancia de los requisitos

Un mal manejo de los requisitos puede causar:
Consecuencias
  • Retrasos en la entrega del proyecto.
  • Incremento de costos.
  • Insatisfacción del cliente.
  • Fallas en el sistema.

Características de un buen requisito

Ejemplo
CaracterísticaSignificado
ClaroNo da lugar a interpretaciones distintas.
CompletoContiene toda la información necesaria.
ConsistenteNo contradice a otros requisitos.
VerificableSe puede comprobar si se cumplió o no.

Tema 2 — Tipos de Requisitos

Requisitos de Negocio

Definición: Describen los objetivos de la organización.
Ejemplo
RN-01 Aumentar las ventas en línea en un 30%.

Requisitos de Usuario

Definición: Describen lo que los usuarios necesitan realizar con el sistema.
Ejemplo
RU-01 El cliente debe poder consultar el estado de su pedido.

Requisitos Funcionales

Definición: Describen las funciones específicas que el sistema debe realizar.
Ejemplos
  • RF-01 Registrar usuarios.
  • RF-02 Generar reportes.
  • RF-03 Procesar pagos.
  • RF-04 Consultar inventario.

Requisitos No Funcionales

Definición: Describen cómo debe funcionar el sistema, no qué hace.
Ejemplos
CategoríaEjemplo
RendimientoEl sistema debe responder en menos de 2 segundos.
SeguridadLas contraseñas deben estar cifradas.
DisponibilidadEl sistema debe estar disponible el 99.9% del tiempo.
UsabilidadLa interfaz debe ser fácil de usar para personas mayores.

Restricciones

Definición: Limitan la forma en que se puede desarrollar el sistema.
Ejemplos
  • REST-01 Utilizar MySQL como base de datos.
  • REST-02 Desarrollar la aplicación en Java.
  • REST-03 Cumplir con la Ley de Protección de Datos.

Tema 3 — Proceso de Ingeniería de Requisitos

1. Elicitación de Requisitos

Definición: Recopilar información de los interesados (clientes, usuarios, expertos).
Técnicas utilizadas
  • Entrevistas Conversaciones directas con el cliente.
  • Encuestas Formularios para recopilar opiniones masivas.
  • Observación Ver cómo trabaja el usuario en su entorno.
  • Talleres Sesiones grupales para llegar a acuerdos.
  • Prototipos Mostrar bocetos para que el cliente valide ideas.

2. Análisis de Requisitos

Definición: Estudiar y organizar los requisitos obtenidos para detectar conflictos, ambigüedades o requisitos incompletos.

3. Especificación de Requisitos

Definición: Documentar formalmente los requisitos en un documento oficial que sirva de referencia para todo el equipo.

4. Validación de Requisitos

Definición: Verificar que los requisitos documentados sean correctos, completos y aceptados por el cliente antes de comenzar el desarrollo.

5. Gestión de Requisitos

Definición: Controlar los cambios en los requisitos durante todo el ciclo de vida del proyecto.
Ejemplo — Proceso completo en una app de delivery
  • 1. Elicitación Entrevistar al dueño: ¿Qué necesita la app?
  • 2. Análisis Revisar si los requisitos son viables y sin contradicciones.
  • 3. Especificación Escribir el documento SRS con todos los requisitos.
  • 4. Validación El cliente revisa y aprueba antes de programar.
  • 5. Gestión Si el cliente pide un cambio, se registra y evalúa su impacto.

Tema 4 — Documentación y Especificación de Requisitos

Documento SRS (Software Requirements Specification)

Definición: Es el documento oficial que contiene todos los requisitos del sistema. Sirve como contrato entre el cliente y el equipo de desarrollo.
Estructura básica de un SRS
  • 1 Introducción
  • 2 Objetivos
  • 3 Alcance
  • 4 Requisitos Funcionales
  • 5 Requisitos No Funcionales
  • 6 Restricciones

Casos de Uso

Definición: Representan las interacciones entre los usuarios y el sistema. Describen qué hace el usuario y cómo responde el sistema.
Ejemplo — Caso de uso: Consultar pedido
CampoDescripción
ActorCliente
AcciónEl cliente ingresa su número de pedido.
Respuesta del sistemaEl sistema muestra el estado actual del pedido.

Historia de Usuario

Definición: Formato ágil para describir un requisito desde la perspectiva del usuario. Sigue la estructura:
Como [usuario], quiero [acción], para [beneficio].
Ejemplo
HU-01 Como cliente, quiero consultar mis pedidos, para conocer el estado de mis compras.
